# **🍐 Wave Messenger 設計・技術仕様書**

## **1\. プロジェクト概要**

本プロジェクトは、Holepunch (Pear Runtime) の完全分散型・P2P技術を活用し、中央集権的なサーバーに依存せずに「LINE」と同等の利便性を提供するシンプルかつ高セキュアなメッセンジャーアプリを開発する。

### **1.1 コアコンセプト**

* **完全P2P通信:** メッセージとメディアは端末間で直接送受信され、中央サーバーには一切保存されない。  
* **高セキュリティ:** エンドツーエンド暗号化(E2EE)を標準搭載。アカウント登録（電話番号やメール）を不要とする。  
* **シンプルさの追求:** 複雑なタイムラインやニュース機能は排除し、純粋なコミュニケーション機能に特化する。  
* **多言語対応:** グローバルな利用を想定し、初期段階から多言語対応(i18n)を組み込む。

## **2\. 設計仕様書 (Design Specification)**

### **2.1 機能要件**

実装する機能は以下に限定する。

1. **チャット機能:** テキストメッセージのリアルタイム送受信、未読/既読管理。  
2. **スタンプ・絵文字:** アプリ内蔵スタンプ、および分散ファイルシステム(Hyperdrive)を利用したスタンプ画像の共有。OS標準の絵文字サポート。  
3. **音声・テレビ電話機能:** 1対1の高音質・高画質P2P通話。  
4. **複数ユーザー同時通話:** メッシュネットワークによるグループ音声/ビデオ通話。  
5. **ローカル電話帳:** 公開鍵（ID）またはQRコード読み取りによる「友達追加」。データは端末内にのみ保存。  
6. **プッシュ通知（ハイブリッド）:** アプリがバックグラウンドにある際の着信・受信通知（※メタデータのみを通知サーバー経由で送信）。  
7. **多言語UI:** 日本語、英語をはじめとする複数言語の切り替え。

### **2.2 画面構成 (UI/UX)**

React (+ Tailwind CSS) を使用し、直感的で馴染みのあるUIを提供する。

* **Splash & Init画面:** アカウント（キーペア）の自動生成、初期プロファイル（名前、アイコン）の設定。  
* **ホーム（電話帳）タブ:** 友達リスト、自分のプロフィール設定、QRコード表示/スキャン画面。  
* **トーク一覧タブ:** 過去のチャットルーム一覧、未読バッジ表示。  
* **トークルーム画面:** 吹き出し形式のメッセージ表示、スタンプ入力パネル、通話発信ボタン。  
* **通話画面:** フルスクリーンのビデオ表示、ミュート/カメラオフ切り替えトグル、通話終了ボタン。複数人の場合はグリッド表示。

## **3\. 技術仕様書 (Technical Specification)**

### **3.1 システムアーキテクチャ**

* **実行環境:** Pear Runtime (Desktop: Windows/macOS/Linux, Mobile: iOS/Android)  
* **UIフレームワーク:** React.js, Tailwind CSS  
* **多言語対応:** i18next, react-i18next  
* **P2P基盤 (Holepunch):**  
  * ネットワーク検索・接続: Hyperswarm  
  * データベース (Key-Value): Hyperbee  
  * メッセージログ (Append-only): Hypercore, Autobase  
  * ファイル共有: Hyperdrive

### **3.2 データ構造とストレージ設計**

#### **データ保存の基本原則**
中央データベースが存在しないため、チャット履歴、通信ログ、連絡先、メディア等の**すべてのデータは、話者同士のデバイスのローカルストレージ（Corestore）にのみ保存される。** デバイス間でP2P同期されるが、中央法人が運用するサーバーには一切の履歴は保存されない。

#### **A. 電話帳 (Hyperbee)**

端末ローカルに保存されるB-Treeデータベース。

* **Key:** contact:\<PublicKey\>  
* **Value:** { name: "Jami", avatarUrl: "hyperdrive://...", status: "online", addedAt: 167888... }

#### **B. チャットルーム (Autobase \+ Hypercore)**

複数人が書き込むチャットルームの履歴を整合性を持って統合する。

* 各ユーザーは自分自身の発言を書き込む Hypercore (ログ) を持つ。  
* Autobase が参加者全員の Hypercore を読み込み、時系列順にマージしてチャットのUIに渡す。  
* **Message Payload:**
  ```json
  {
    "type": "text|sticker|file",
    "text": "こんにちは",
    "timestamp": 167888...,
    "fileHash": "sha256:...",  // 転送ファイルがある場合のハッシュ
    "messageId": "seq-..."     // 自律的なメッセージID
  }
  ```
* **ファイルとの整合性確認:**
  * 転送された各ファイルは、そのコンテンツハッシュ（BLAKE3/SHA-256）によって履歴と結びつけられる。
  * ファイル側のメタデータにも、それがどのメッセージ（messageId）に付随するものかを記録し、後から「履歴上のこのメッセージに関連するファイルであること」を双方向で検証できる設計とする。

### **3.3 ネットワーク設計 (通信フロー)**

#### **3.3.1 テキスト・スタンプの送受信**

1. ユーザーAがメッセージを入力。  
2. アプリがユーザーAのローカル Hypercore にメッセージを追記 (Append)。  
3. Hyperswarm を通じて、ユーザーBにブロックの更新が通知される。  
4. ユーザーBの端末がP2Pで差分データをダウンロードし、UIを更新する。

#### **3.3.2 音声・ビデオ通話 (WebRTC over Hyperswarm)**

Hyperswarm を「シグナリングサーバー」の代わりとして利用する。

1. **シグナリング:** AからBへ電話をかける際、Hyperswarm の暗号化ソケットを通して WebRTC の SDP (Offer/Answer) ICE Candidate を交換する。  
2. **ピア接続:** WebRTC の RTCPeerConnection が確立される。  
3. **ストリーミング:** 以降の音声・映像データは UDP ベースで直接端末間を流れる。複数人の場合は、全員が相互に接続する「フルメッシュ構成」とする（参加人数は最大8人程度を推奨）。

#### **3.3.3 QRコードによるアドレス交換 (Handshake via QR Code)**

物理的に対面している状況、または外部媒体を通じて安全にアドレス（公開鍵）を交換するためのプロトコル。

1. **QRコードの生成:** 自分の公開鍵（Ed25519）を Base64 または特定のカスタムURL（wave://join/<PublicKey>）形式でQRコード化して表示する。
2. **スキャンと検証:** 相手がアプリ内スキャナーで読み取ると、公開鍵が抽出される。
3. **初期ハンドシェイク:** スキャナー側が、抽出した公開鍵をターゲットとして Hyperswarm 上で検知（Discovery）を開始。接続が確立した時点で「友達追加」の承認フローが実行される。
4. **属性交換:** 相互に承認されると、名前やアバターURL（Hyperdriveパス）などの最小限のプロファイル属性がローカルの Hyperbee に登録される。

### **3.4 セキュリティ設計**

* **エンドツーエンド暗号化 (E2EE):** Hyperswarm の接続は標準で Noise Protocol Framework を使用し、自動的に暗号化される。  
* **アイデンティティ:** アプリ初回起動時に Ed25519 暗号キーペアを生成。公開鍵（32バイト）がそのままユーザーIDとなり、秘密鍵はデバイス外に絶対に出ない。  
* **データ保護:** ローカルストレージ内の Hypercore データも暗号化して保存する機構を設ける。

### **3.5 プッシュ通知システム (ハイブリッド設計)**

P2Pの弱点である「アプリ終了時の着信」をカバーするため、メタデータのみを扱う軽量リレーサーバーを構築する。

* **前提:** 受信側はアプリ起動時に、OS (Apple/Google) から取得した「デバイストークン」と自分の「公開鍵」のペアをリレーサーバーに登録しておく。  
* **送信フロー:**  
  1. AがBに電話をかける（またはメッセージを送る）。BへのP2P接続がタイムアウトする。  
  2. Aはリレーサーバーに POST /notify { targetPublicKey: "B\_PUB\_KEY", type: "call\_ring" } を送る。  
  3. リレーサーバーは DBから B のデバイストークンを引き、FCM/APNs に「着信通知」を要求する。  
  4. Bのスマホが鳴る。Bが応答してアプリを開くと、即座に Hyperswarm 経由のP2P通信に切り替わり、通話が開始される。  
* **プライバシー担保:** サーバー側には「誰が誰にどんなメッセージを送ったか」は記録されず、通知トリガーのみが通過する。

### **3.6 大容量データ転送設計 (Large File Transfer)**

数GBクラスのデータを高速かつ堅牢に送受信するため、Hyperdriveの分散ファイルシステム機能を拡張利用する。

*   **チャンク分割とMerkle Tree:** 
    *   ファイルは自動的に数MB単位の「チャンク」に分割される。
    *   各チャンクはハッシュ化され、Merkle Tree（ハッシュツリー）によってデータの完全性が保証される。受信側は破損したチャンクのみを再送要求できる。
*   **高速並列転送:** 
    *   Hyperswarmの複数の接続（ピア）を同時に利用し、異なるチャンクを並列でストリーミング配信することで、帯域幅を最大限に活用する。
*   **レジューム（中断・再開）機能:** 
    *   Hypercoreのインデックス管理により、通信が途切れた際も「どのブロックまで受信したか」を正確に把握。次回接続時に未取得分から即座に再開する。
*   **スパース・同期 (Sparse Sync):** 
    *   受信側はファイル全体の完了を待たずに、ストリーミング再生（動画など）のように必要な部分から優先的にダウンロードを開始できる。

### **3.7 ファイル預かりサービス (Paid File Hosting - GigaFile style)**

将来的なアップデートとして、P2Pの「双方がオンラインである必要性」を解消するための「ファイル預かり（ストレージサーバー）機能」を実装する。

*   **サービス概要:** 
    *   送信者がファイルをサーバーにアップロードし、特定の期間（例：7日間〜）データを保持。受信者は送信者がオフラインでもダウンロード可能。
    *   GigaFileのように大容量（1契約あたり最大数百GB〜）の預かりに対応。
*   **有料プランとビジネスモデル:** 
    *   **無料枠:** 短期間（3日間）かつ制限された容量の転送。
    *   **有料プラン:** 長期保存、保存容量の拡張、パスワード保護、高速アップロード優先権などを提供。
    *   サブスクリプションまたは従量課金制を採用。
*   **認証と決済 (Store Compliance):** 
    *   Apple App Store / Google Play Storeの規約に準拠するため、有料機能の利用には「ユーザーアカウント認証」を必須とする。
    *   決済は StoreKit (iOS) / Google Play Billing (Android) を介したアプリ内課金として実装する。
*   **セキュアな一時保存:** 
    *   サーバーに保管されるデータは、送信者の秘密鍵で生成された一時的な共有鍵で暗号化される。サーバー運営者であっても内容を照会できない独自のE2EEストレージ方式を採用する。

## **4\. 開発・実装フェーズ**

* **Phase 1: コア機能実装 (CLI/Desktop版)**  
  * Pear Runtimeの初期化。  
  * Hypercoreによる1対1のテキストチャット同期。  
  * React UIの基本的な組み込みと多言語対応(i18n)基盤の整備。  
* **Phase 2: リッチメディアと通話**  
  * WebRTCとHyperswarmを統合した1対1の音声・ビデオ通話。  
  * Hyperdriveを用いたスタンプ機能の実装。  
* **Phase 3: モバイル最適化とグループ機能**  
  * Pear Mobileを使ったAndroid/iOSビルド設定。  
  * リレーサーバーの実装とプッシュ通知(FCM/APNs)の連携。  
  * Autobaseを利用した複数人チャット・通話への拡張。
* **Phase 4: 大容量転送最適化と収益化基盤**
  * Hyperdriveのチャンク転送プロトコルの高速化チューニング。
  * ファイル預かりサーバー（Storage Node）の開発とデプロイ。
  * アプリ内課金 (Internal Purchase) システムとユーザー管理基盤の統合。
  * 多言語によるヘルプセンターとサポート機能の拡充。
